# PROMPT-0036 — T-024 (SPEC-0007): registry de objetos + formato de mapa v1 + loader · 2026-07-05

## Pedido (resumo fiel do CD)
Continuação direta da fila da V1 depois do fechamento da F2 (T-023/PROMPT-0035) — "vamos prosseguir agora com o desenvolvimento das próximas", sem alinhamento por task. Primeira da F3: **T-024**, escopo fechado na SPEC-0007 (registry + formato + loader; a CLI com `save-current` fica para a T-025).

## Decisões tomadas (e por quem)
- **Registry de objetos (`packages/shared/src/objects.ts`):** `ObjectDef {id, footprint:{w,h}, collidable}` cobrindo os 4 tipos colidíveis que já existiam no gerador por seed (`pedra`, `arvore`, `caixa`, `muro`) + `bandeira` (decorativa, mesmo footprint dos marcadores de zona de guerra já renderizados). Mesma interface hoje vem do código; quando o backend Django existir (SPEC-0008), objetos salvos pelo sistema usam a mesma forma — só a origem muda.
- **Formato de mapa v1 (`packages/shared/src/mapFile.ts`, `MapFileV1`):** `{version, id, name, author?, w, h, seed?, instances:[{objectId,x,z,rot?,scale?}], zones, spawns, flag}`. `rot`/`scale` existem no tipo mas ainda são ignorados na conversão (reservados — nem o gerador por seed rotaciona/escala props hoje, então não é regressão).
- **`mapFileToGameMap(file)` produz a MESMA estrutura `GameMap`** que `buildMap(w,h,seed)` já produzia — decisão central que manteve o resto do código (colisão, zonas, `zoneAt`/`isWall`, bots) **inalterado**: ninguém downstream precisa saber se o mapa nasceu de seed ou de arquivo curado.
- **`validateMapFile(file)` retorna lista de erros (nunca lança)** — objectId desconhecido, instância/spawn/bandeira fora dos limites, e (só se o resto passou) **flood-fill** via novo `floodFillReachable(map)` em `map.ts` (BFS 4-direções sobre o grid de células livres, confirma que todas formam uma única região conectada). O gerador por seed já garante isso estruturalmente (`isIsolated` força props a nunca se tocarem — nenhuma mudança nele); mapas curados por arquivo, editados à mão, precisavam da checagem explícita.
- **Loader é código só de servidor** (`packages/server/src/mapLoader.ts`, `readFileSync`) — browser não lê filesystem, então a leitura do JSON só existe no lado do servidor; o `MAPS_DIR` resolve a partir de `__dirname` (estável porque o servidor sempre roda via `tsx` direto do source, sem etapa de build/dist). Mapa inválido lança `Error` com a lista de erros concatenada (critério de aceite "erro claro").
- **Sala aceita `mapId` opcional (`ArenaRoom.onCreate`)** — presente = carrega o arquivo curado (`loadMap`), ausente = caminho original por seed **preservado tal e qual** (nada quebra para quem não usa mapas curados ainda). `ArenaState` ganhou `mapId: string = ""` (vazio = seed) — é assim que cliente/bots sabem qual caminho seguir.
- **Mapa curado não cabe em reconstrução por seed** — como o cliente/bots não podem ler `maps/*.json` do disco do servidor, o servidor manda o `MapFileV1` completo por mensagem (`client.send("map_data", ...)`) uma única vez no join (spec já previa isso — "mapas pequenos"); cliente e bots chamam `mapFileToGameMap` localmente. Caminho por seed (sem `mapId`) continua zero-tráfego, como sempre.
- **Spawns do mapa curado substituem os 8 cantos/meios-de-borda genéricos** — `file.spawns` vira a lista usada em `onJoin`, permitindo ao curador decidir onde os jogadores nascem.
- **`BOT_MAP_ID` (env var, mesmo padrão de `BOT_BOSS`/`BOT_PROFILE`)** — só o bot que cria a sala manda o `mapId`; permite QA/smoke de um mapa curado sem esperar a CLI da T-025.
- **1 mapa de exemplo (`maps/arena-teste.map.json`)** criado à mão para provar o pipeline ponta a ponta — não conta como os "2 mapas curados" do critério de aceite geral (que depende de `save-current`, T-025), é só fixture de verificação.

## Regras que nascem daqui
- **`mapFileToGameMap` é o único ponto de conversão formato→`GameMap`** — qualquer novo consumidor (servidor, cliente, bots, CLI da T-025) usa essa função; nunca duplicar a lógica de marcar paredes por footprint.
- **Todo objeto de mapa nasce como uma entrada em `OBJECT_DEFS`** (`packages/shared/src/objects.ts`) — mesmo padrão de "efeito novo = 1 entrada de dados" já estabelecido para VFX (T-022); objeto novo = 1 `ObjectDef` (footprint + colide ou não), nunca lógica ad-hoc espalhada.
- **Mapa curado (`mapId` setado) nunca reconstrói por seed no cliente/bots** — sempre espera `map_data`; qualquer código futuro que leia `state.mapW/mapH/mapSeed` para desenhar o mundo precisa checar `state.mapId` primeiro (o padrão já está em `main.ts`/`bot.ts`, replicar em qualquer novo consumidor).
- **Validação de mapa nunca lança direto** — `validateMapFile` sempre retorna array de erros; decidir o que fazer com eles (lançar com mensagem clara, recusar salvar, etc.) é responsabilidade de quem chama.
